# Internationalization (i18n)

> When to load: adding/translating UI copy, formatting dates/numbers/currency, adding a
> locale, touching locale routing or the `[locale]` segment, per-locale SEO
> (hreflang/sitemap/`<title>`), the `LanguageSwitcher`, or the navigation helpers. The _why_ behind the approach is in
> [DECISIONS.md](DECISIONS.md); this is the how.

## At a glance

- **Library:** [`next-intl`](https://next-intl.dev) (`^4.13.1`) — the App-Router-native i18n
  library. Wired into Next via `createNextIntlPlugin` in `next.config.ts`.
- **Locales:** `en` (default) + `es`. The single source of truth is
  `apps/web/src/i18n/routing.ts` (`routing.locales`).
- **Strategy — Mode A (`[locale]` path routing):** the locale lives in the URL as the first
  path segment. The whole page tree is under `app/[locale]/`. `localePrefix: "as-needed"` keeps
  the **default locale unprefixed** (`/`, `/login`, `/dashboard`) and prefixes the rest (`/es`,
  `/es/login`). This is what keeps `[locale]` routes **statically prerendered** under
  `cacheComponents`/PPR (a cookie-based locale would force every route dynamic).
- **Coverage is FULL-SURFACE** (2026-07-16 — supersedes the launch-era
  "partial by design"): every page and component under `[locale]` is translated — the primary
  journey, `/account` (all cards), `/admin` + `/admin/audit`, `/organization` (+ the switcher and
  accept-invitation), and all demo/scaffold routes (`/posts`, `/uploads`, `/search`, `/billing`,
  `/premium`, `/state`, `/observability`), plus the `[locale]` error boundary. Deliberate
  English-only remainders: the `not-found` pages and `loading.tsx` (Next passes them no `params`,
  so they can't `setRequestLocale` — the documented framework exception), the audit-event labels
  from `lib/audit-format` and server-action/API error strings (server-side values, not UI copy),
  and the delete-account confirm phrase `delete my account` (a typed literal compared verbatim).
  Partial coverage remains SAFE for adopters: an un-wired page just keeps its literal string.
- **No new env, no new service, no CSP change.** i18n is same-origin static assets; the message
  catalogs ship in the JS bundle (code-split per locale).

## How routing works

The locale is negotiated + rewritten in the proxy (`apps/web/src/proxy.ts`), then read back out
of the `[locale]` route segment on the server.

1. **`proxy.ts` composes the auth gate with next-intl routing.** `createMiddleware(routing)`
   (`handleI18nRouting`) negotiates the locale, internally rewrites unprefixed default-locale
   paths to the `[locale]` segment, 307-redirects `/en/*` → the unprefixed form, and serves
   `/es/*`. The optimistic cookie-only auth gate runs **first**, on the **locale-stripped path**
   (`stripLocale`), so `/es/dashboard` gates exactly like `/dashboard`; its redirect targets are
   kept locale-correct via `localePrefix()` (an `/es` visitor lands on `/es/login`). A redirect
   response needs no locale rewriting, so the gate short-circuits before handing off to next-intl.
2. **The matcher runs on (almost) everything:** `'/((?!api|_next|_vercel|.*\\..*).*)'`. The
   `.*\..*` clause already excludes the **dotted** metadata routes (`sitemap.xml`, `robots.txt`,
   `manifest.webmanifest`). The **dot-less** file-metadata routes (`opengraph-image`,
   `twitter-image`, `icon`, `apple-icon`) live at the app **root**, outside `[locale]`, and would
   be caught by the broad matcher and rewritten under the default locale → 404. A
   `METADATA_SEGMENTS` guard at the top of `proxy()` `return NextResponse.next()`s for them.
3. **The document shell moved under `[locale]`.** `app/layout.tsx` is now a **bare passthrough**
   (`return children`) that owns only the `globals.css` import (so the stylesheet applies to the
   whole tree, including the non-localized routes and the root `not-found`).
   `app/[locale]/layout.tsx` is the real document layout: `<html lang={locale}>` + `<body>` + the
   app providers + `generateStaticParams` + `generateMetadata`. `<html lang>` flips per locale.
4. **Invalid locales 404.** The `[locale]` layout validates `params.locale` against
   `routing.locales` with `hasLocale` and calls `notFound()` otherwise (defense-in-depth — the
   proxy already normalizes unknown prefixes to the default).

## The i18n plumbing

| File | Role |
| --- | --- |
| `apps/web/src/i18n/routing.ts` | **Source of truth:** `defineRouting({ locales, defaultLocale, localePrefix })` + the `Locale` type. Consumed everywhere. |
| `apps/web/src/i18n/request.ts` | Per-request config (`getRequestConfig`): validates the requested locale, **dynamically imports** `messages/<locale>.json` (code-split — only the active catalog ships). |
| `apps/web/src/i18n/navigation.ts` | `createNavigation(routing)` → locale-aware `Link`, `redirect`, `usePathname`, `useRouter`, `getPathname`. Import these as `@/i18n/navigation` instead of the `next/*` primitives. |
| `apps/web/messages/{en,es}.json` | The message catalogs (one file per locale). |
| `apps/web/next.config.ts` | `createNextIntlPlugin("./src/i18n/request.ts")` → `withNextIntl`, wrapped **innermost**: `withSentryConfig(withNextIntl(nextConfig), …)`. |
| `apps/web/src/app/[locale]/layout.tsx` | Mounts `<NextIntlClientProvider>` (so client components get the active locale + messages via context). |

`pnpm-workspace.yaml` pins next-intl's optional transitive native build scripts (`@swc/core`,
`@parcel/watcher`) to `allowBuilds: false` — they're prebuilt/unused, so there's nothing to compile.

Copy lives in `apps/web/messages/{en,es}.json`, grouped into namespaces. Current layout:

| `Dashboard.{nav,userMenu,home,impersonation}` | The `(dashboard)` shell (nav, user menu, dashboard page, impersonation banner). |
| `Notifications` | The gated `/notifications` realtime demo (A22): feed, send/mark-read buttons, unread count (ICU plural), stream-status badges, toast title. |
| `Account.{page,avatar,name,email,password,twoFactor,passkeys,sessions,privacy,delete}` | The `/account` surface — page chrome + all nine cards. |
| `Admin.{page,role,ban,impersonate,audit,pagination}` | `/admin` + `/admin/audit` + the per-row controls; `pagination` is shared by both keyset navs. |
| `Organization.{page,manager,members,invite,invitations,settings,create,switcher,acceptPage,accept}` | `/organization` (+ all cards), the header OrgSwitcher, and `/accept-invitation/[id]` (server page + client island). |
| `Posts.{page,composer,form,item,list,skeleton,stats}` | The `/posts` PPR showcase (shell, composer, create form, rows, feed states, cached count — ICU plural). |

| `Error` | The `[locale]` route-segment error boundary. |

- **Read messages** with `useTranslations("Namespace")` (client) or
  `await getTranslations({ locale, namespace })` (server). Both `en.json` and `es.json` carry the
  **same key tree** (`es` is real Spanish, not a stub).
- **ICU interpolation** works out of the box: `t("home.signedInAs", { email })`,
  `t("signup.sentDescription", { email })`.
- **Rich text** via `t.rich` when a value embeds markup — the impersonation banner uses
  `"Impersonating <bold>{email}</bold>"` with a `bold` render function.
- **Keep `en` values BYTE-IDENTICAL to the previous hardcoded literal** when you extract a string
  (down to the real `…` ellipsis and curly `'`). The E2E suite runs against the **default,
  unprefixed (en)** locale, so its text/role selectors match the message value. Before renaming
  any wired `en` string, grep the E2E tree for `getBy(Role|Text|Label)` uses of it.
- **Partial coverage is safe** (relevant when adapting the starter): a page that never calls
  `useTranslations`/`getTranslations` just renders its hardcoded English — there's no
  missing-key error. The shipped app is full-surface, but a new page you add works
  before you wire it.

## Formatting dates, numbers & currency (`useFormatter`, A30)

Messages cover *copy*; **values** (dates, counts, prices) format through the active locale
with `useFormatter` — not `toLocaleDateString()` / hand-rolled `Intl.*` instances, which
ignore the negotiated locale and drift between call sites. All snippets below are
type-check-verified against the installed `next-intl@4.13.1`.

- **Client components:** `const format = useFormatter()` (from `next-intl`), then
  `format.dateTime(date, { dateStyle: "medium" })` ·
  `format.number(price, { style: "currency", currency: "USD" })` ·
  `format.relativeTime(date)` ("2 hours ago" / "hace 2 horas") ·
  `format.list(["read", "write"])` ("read and write" / "read y write") ·
  `format.dateTimeRange(start, end, { dateStyle: "short" })`.
- **Server:** `useFormatter` also works in **non-async** server components (same import).
  Async code — `generateMetadata`, Server Actions, route handlers — uses
  `const format = await getFormatter()` from `next-intl/server`; outside request scope
  pass the locale explicitly, `getFormatter({ locale })` (the `getTranslations({ locale })`
  pattern from SEO above).
- **Named formats — define once, use everywhere.** Return a `formats` object from
  `getRequestConfig` in `request.ts`:

  ```ts
  return {
    locale,
    messages: (await import(`../../messages/${locale}.json`)).default,
    formats: {
      dateTime: { short: { dateStyle: "medium", timeStyle: "short" } },
      number: { price: { style: "currency", currency: "USD" } },
    },
  };
  ```

  Call sites then say `format.dateTime(d, "short")` / `format.number(n, "price")`
  (a third argument can still override per-call). The bare `<NextIntlClientProvider>`
  in the `[locale]` layout **auto-inherits `formats` (and `timeZone`/`now`) from the
  request config** when rendered from a server component (verified in 4.13.1's
  provider source), so client and server share the one definition with no extra wiring.
- **ICU-embedded values** format inside message strings with no `useFormatter` call:
  `"price": "Costs {price, number, ::currency/USD}"` → `t("price", { price: 42 })`,
  and `{when, date, medium}` for dates (`intl-messageformat` number/date skeletons ride
  along with next-intl). Prefer this when the value sits inside translated copy; use
  `useFormatter` for standalone values (table cells, timestamps, totals).
- **The `timeZone` gotcha (read before shipping absolute times).** Without a `timeZone`
  in the request config, dates SSR in the **server's** zone but hydrate in the
  **user's** zone → mismatched markup; next-intl raises `ENVIRONMENT_FALLBACK`
  ("Consider adding a global default to avoid markup mismatches") in dev. Set an
  explicit `timeZone` in `request.ts` when server-rendering absolute times. Same class:
  `relativeTime` without a `now` argument falls back to the current time per
  environment — pass `now` explicitly (or configure a global `now`) in prerendered
  output. **The shipped `request.ts` now carries this block**: a
  `formats.dateTime.short` named format (`medium`+`short`, the `toLocaleString()` analog),
  a `formats.dateTime.dateOnly` named format (`medium` date, no time — for passkey-added /
  invitation-expiry / renewal dates where the clock is noise), and a global
  `timeZone: "UTC"` (deterministic — a real app that wants the user's local time overrides
  it per request). **Every date the UI renders goes through these named formats**:
  the notifications feed, `/admin/audit` timestamps (previously a hand-rolled en-US `Intl`
  instance — the named format + global UTC zone render identically for `en` and localize
  for the rest), the sessions card, post rows, the uploads list (`short`), and the
  passkeys / pending-invitations / billing-renewal dates (`dateOnly`). No `toLocale*`
  call sites remain (the one deliberate exception: `server/actions/notification.ts`
  generates a test-notification BODY with `"en-US"` — a data value, not UI copy). A
  global `now` is still deferred — no route server-renders a `relativeTime` yet; add it
  alongside the first one.

## The client-vs-server navigation split (the crux)

Swapping `next/link` + `next/navigation` for `@/i18n/navigation` is what keeps the active locale
**sticky** across navigation (a plain `<Link href="/dashboard">` from an `/es` page would drop
back to the default locale). But client and server components consume it differently:

- **Client components (`"use client"`) — a drop-in swap.** `Link`/`useRouter` from
  `@/i18n/navigation` read the active locale from `NextIntlClientProvider` (client context), so
  changing the import is the whole change. (All the auth forms, user-menu, org-switcher, account
  cards, `error.tsx`, etc.)
- **Server pages/layouts that render the i18n `<Link>` need `setRequestLocale`.** The server
  `<Link>` reads the locale via `headers()`, which under `cacheComponents` is "uncached data
  accessed outside `<Suspense>`" → a **hard prerender failure**. Fix: call
  `setRequestLocale(locale)` (from `next-intl/server`, using `params.locale`) at the **top of
  every server page/layout** before rendering translations or an i18n `<Link>`. This is *also* the
  static-rendering enabler — it's what keeps the route prerendering, not a dynamic read. It's
  called in **every server page** under `[locale]` (plus the `(dashboard)` + `(auth)`
  layouts). Two worked variants: a **synchronous** page (the `/posts` + `/state` PPR shells)
  reads params via React's `use(params)` before `useTranslations`; and an async Suspense child
  that renders **outside the shell pass** (the `/posts` ComposerCard / PostStats — request-time
  streaming or post-`updateTag` regeneration) takes the locale as a **prop** and re-anchors it
  itself with its own `setRequestLocale` call rather than relying on the shell's.

**Three deliberate exceptions that stay on `next/navigation`:**

- **`redirect` is NOT a drop-in.** next-intl's `redirect` takes a `{ href, locale }` object (not a
  bare string) and, as tsc resolves it under `moduleResolution: bundler`, it doesn't give the
  `never`-based control-flow narrowing that `next/navigation`'s `redirect` does — so
  `if (!session) redirect(...)` stops narrowing `session`. The four `(dashboard)` backstop
  `redirect("/login")` calls stay on `next/navigation` (they're proxy+layout-guarded backstops, so
  the marginal stickiness loss doesn't matter).
- **`login`/`signup` `router.push(redirectTo)` stays on `next/navigation`** — the **double-prefix
  trap.** `redirectTo` is a proxy-built **full** locale path (`/es/login?redirectTo=/es/dashboard`);
  the i18n router would *prepend* the active locale → `/es/es/dashboard`. Pushing the literal
  string via `next/navigation` produces the correct `/es/dashboard`.
- **`not-found.tsx` stays on `next/link`.** Next doesn't pass `params` to `not-found`, so it can't
  `setRequestLocale` → its trivial `href="/"` uses the framework `Link` (resolves to the default
  locale, fine for a 404). The root `not-found.tsx` (own `<html>`, outside the provider) likewise.

## SEO: metadata, hreflang & sitemap

- **`generateMetadata` reads the locale EXPLICITLY, so it's PPR-safe.**
  `getTranslations({ locale, namespace })` takes the locale from `params` (not `headers()`), so
  every `[locale]/*` route still prerenders. `setRequestLocale` is **not** needed inside
  `generateMetadata` (only in the page-body render path). A route exports **either** `metadata`
  **or** `generateMetadata`, never both.
- **`localizedAlternates({ locale, href })`** (`apps/web/src/lib/i18n-metadata.ts`) builds the
  `alternates` block — the self-referential `canonical` plus the `en`/`es`/`x-default` `languages`
  map — for one indexable page. It calls `getPathname({ locale, href })`, a **pure** computation
  over the routing config (no request scope), so it's safe in both `generateMetadata` and the
  sitemap. Values are **relative pathnames**; the root layout's `metadataBase` (`new URL(siteUrl)`)
  resolves them to absolute URLs in the emitted `<link rel="alternate">` tags.
- **OpenGraph locale codes** come from an `OG_LOCALES` map (`{ en: "en_US", es: "es_ES" }`) in the
  `[locale]` layout → `og:locale` + `og:locale:alternate`.
- **`sitemap.ts` lives at the ROOT**, outside `[locale]`, so there's no `metadataBase` to resolve
  relatives — it prefixes `siteUrl` itself. It emits one `<url>` **per locale** for each of
  `PUBLIC_PATHS` (`/`, `/login`, `/signup`, `/forgot-password` — the real public surface only), each
  carrying the full `<xhtml:link>` hreflang set. It's dynamic (`ƒ`) because `lastModified` reads
  `new Date()`.
- **Which pages get what** (primary journey; demo/scaffold excluded): login/signup/forgot-password
  get title **+** hreflang; the landing page gets hreflang only (brand-default title);
  reset-password/goodbye/dashboard get title only (token-gated / noindex / auth-gated → no
  hreflang). The `metadataBase property … not set` build warning for the **root-level** OG/twitter
  image routes is pre-existing (the passthrough root has no `metadataBase`) and cosmetic — the
  `[locale]` pages do carry it (curl-verified absolute hreflang/canonical).

## The LanguageSwitcher

`apps/web/src/components/language-switcher.tsx` — a `DropdownMenu` radio group in the top-right
control cluster (beside `ThemeToggle`) on the landing page, the `(auth)` shell, and the
`(dashboard)` header.

- Selecting a locale calls `router.replace(pathname, { locale })` from `@/i18n/navigation`;
  `usePathname()` (also from `@/i18n/navigation`) returns the locale-**stripped** path and the
  router re-prefixes per `localePrefix`. `useLocale()` reads the active locale from context.
- **It deliberately does NOT read `useSearchParams`** — that would force the statically-rendered
  landing + `(auth)` shells to add a `<Suspense>` boundary under `cacheComponents`. A language
  toggle doesn't need to preserve the query string, so it's dropped on switch.
- The radio **items** are endonyms (`LOCALE_NAMES` → "English"/"Español"), the same in every
  locale, so they're **not** translated. Only the sr-only **trigger label** is localized
  (`LanguageSwitcher.label`) — so a test driving it across a switch must match a locale-agnostic
  selector (e.g. `name: /Change language|Cambiar idioma/`).

## Add a locale (recipe)

To add, say, French (`fr`):

1. **`apps/web/src/i18n/routing.ts`** — add `"fr"` to `locales`.
2. **`apps/web/messages/fr.json`** — copy `en.json` and translate every value (keep the key tree
   identical).
3. **`OG_LOCALES`** in `apps/web/src/app/[locale]/layout.tsx` — add `fr: "fr_FR"`.
4. **`LOCALE_NAMES`** in `apps/web/src/components/language-switcher.tsx` — add `fr: "Français"`.

Everything else derives from `routing.locales`: `generateStaticParams` prerenders it, the sitemap +
`localizedAlternates` emit its hreflang, the proxy negotiates it, `request.ts` code-splits its
catalog. No routing, proxy, or component wiring changes.

## Testing i18n

- **Unit (`apps/web/src/lib/i18n-metadata.test.ts`, node env).** Testing a module that reaches
  `@/i18n/navigation` in the **node** vitest env needs two things in `vitest.config.ts`: (a) alias
  stubs for `next/navigation` (`src/test/next-navigation.stub.ts`) + `next/link`
  (`src/test/next-link.stub.ts`), because next-intl's `createNavigation` imports those Next runtime
  modules at load; **and** (b) `test.server.deps.inline: ["next-intl"]` — **without the inline the
  aliases don't apply**, since vitest externalizes `node_modules` and never sees the
  `next/navigation` import inside next-intl. `getPathname` is pure, so the stubs only satisfy the
  ESM link check. Assert on `Metadata["alternates"]` with **whole-object** matchers
  (`toEqual`/`toMatchObject`) to avoid the optional/nullable shape.
- **E2E (`apps/web/e2e/i18n.spec.ts`).** Public/unauth flows only → **DB-free** (not subject to the
  signUp-flake). Covers the switcher flipping the landing en↔es, `<title>` localizing (`/login` vs
  `/es/login`), hreflang `<link>`s in `<head>`, unauth `/es/dashboard` →
  `/es/login?redirectTo=/es/dashboard`, and the `/es/posts` demo chrome rendering in Spanish
  (static-shell assertions, still DB-free). The es sign-in submit check needs `{ exact: true }` —
  "Iniciar sesión" is a substring of the passkey button "Iniciar sesión con una clave de acceso".
  A **signed-in** es spot-check rides `account.spec.ts`'s serial lifecycle (no extra signup
  against the 5/60s limiter): `/es/account` title + chrome plus a Spanish abbreviated-month date
  from the sessions card — proof the named formats localize end-to-end.

## Gotchas (quick reference)

- **Dot-less metadata routes** (`opengraph-image`/`twitter-image`/`icon`/`apple-icon`) need the
  `METADATA_SEGMENTS` proxy guard — the broad matcher would otherwise 404 them under the default
  locale.
- **`redirectTo` double-prefix:** it's already a full locale path from the proxy — push it with
  `next/navigation`, not the i18n router.
- **`setRequestLocale` before render** in every server page/layout that translates or renders an
  i18n `<Link>`, or the build fails under `cacheComponents`.
- **`en` message values are E2E-load-bearing** — keep them byte-identical when extracting.
- **`redirect` from `@/i18n/navigation` breaks control-flow narrowing** — keep guard redirects on
  `next/navigation`.
