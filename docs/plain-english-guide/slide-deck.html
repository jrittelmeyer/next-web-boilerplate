The problem

The feature you want to build is the smallest part of the work

Before any product can take its first customer, a team must build the same invisible foundation every product needs — none of it optional, none of it their specialty:

  • Sign-in and account security — where mistakes make headlines
  • Safe data storage — with backups that provably restore
  • Payments, email, file handling, search — each with brutal edge cases
  • Monitoring, testing, security hardening, deployment — the difference between a demo and a business

Industry name: undifferentiated heavy lifting — realistically 4–6 months of senior engineering re-solved from scratch by almost every project, usually less well. This project is that lifting, done once, done properly.

The thesis

Three things free starter kits almost never have

  • 1 · Verified, not assumed. Every integration was exercised for real — live test payments, real emails, real deploys — with dated proof kept in the repository. Not “wired and hopefully works.”
  • 2 · Every decision on file. A written decision log records why each technology was chosen, what the alternatives were — and which options were evaluated and rejected, with evidence. Nobody ever has to guess “why is it like this?”
  • 3 · Built for the AI era. Structured so AI coding agents work in it as effectively as humans — which is also how it was built, under human sign-off, with machine-enforced quality. The same structure doubles as the best new-hire onboarding you’ve seen.

Feature value · Identity

Enterprise-grade sign-in, owned outright — $0 per user, forever

  • No identity rent, no lock-in. Rented login services charge per active user — success makes them expensive — and hold your user data hostage. Here, accounts live in your own database.
  • Security that survives due diligence: passwords checked against real-world breach databases, two-factor codes, fingerprint/passkey login (unphishable), bot protection, attempt throttling.
  • B2B-ready from day one: teams/organizations with roles and invitations — the structure business customers require.
  • Permissions that bite instantly. Rights are re-checked fresh on every sensitive action — a revoked admin loses power immediately, not minutes later (a real flaw in most systems, documented honestly where one narrow exception remains).
  • Compliance groundwork done: permanent audit trail of admin actions, user data export, session management, safe email-change — the early questions on every security questionnaire, already answered.

Feature value · Payments

Revenue-ready on day one

  • Subscription checkout → money in, recorded correctly. Stripe’s hosted payment page (card data never touches your servers — the lightest compliance posture) feeding a subscriptions record in your own database.
  • The messy parts are the point: duplicate payment notifications handled exactly once; failed renewal cards detected and status-flipped automatically (“dunning” — a leading cause of silent revenue loss); premium content gating worked end-to-end.
  • Fewer support tickets: customers manage cards and cancel through Stripe’s self-service billing portal.
  • No billing zombies: deleting an account automatically cancels its subscription — the classic infuriating loophole, closed.
  • Teams can pay, not just individuals: one team subscription entitles every member — B2B-ready — with ownership rules engineered so no member’s account deletion can ever cancel the team’s billing.
  • All of it proven with real test transactions, including time-travel simulation of renewal failures months out.

Feature value · Email

Email that reaches inboxes, not spam folders

  • The deliverability plumbing is the hard part — the DNS-level standards (SPF/DKIM/DMARC) that make providers trust your domain. Documented as a recipe and executed against a real domain, verified 2026-07-14.
  • Professional templates, safely built. Emails are written with the same type-checked component system as the app — no fragile hand-coded email HTML — with a plain-text version on every send (a spam-filter best practice most kits skip).
  • Every account flow wired: verification, password reset, welcome, safe email change.
  • Runs fine with no email provider at all — sign-up still works; team invites fall back to a copyable link. Nothing blocks evaluation.

Feature value · Search, uploads & background work

Big-product capabilities, small-product operating costs

  • Instant, typo-tolerant search (Meilisearch) — the search experience users expect from Amazon, self-hosted: no per-query rent, unlike the famous alternative (Algolia). Index settings are versioned in the repository like code.
  • Safe file uploads — browsers upload straight to cloud storage (your servers never carry the load), every file tracked in the database, deletion engineered so a half-failure can never orphan a file.
  • A retrying background-job queue — slow work (emails, billing calls) happens off the customer’s clock and retries on failure. The signature decision: the queue lives inside the existing database — zero new services to run, and if the worker is down, jobs safely accumulate and drain later.
  • The pattern to notice: one boring, excellent database also powers the job queue, live updates, and rate limiting — four systems’ capability, one system’s operating burden.

Feature value · Live updates

A live, modern feel — without renting another service

  • Notifications appear the instant they happen — no refresh — with an always-correct unread badge. Shipped as a complete worked example.
  • The typical cost of this feature is a new vendor or a new server cluster. Here it’s built on capabilities the database and every browser already have — zero new infrastructure, zero new dependencies.

  • Limits documented honestly — including where certain hosting styles constrain it, with named fallbacks and the scale-up path written down.

Feature value · Data

The unrecoverable part is the most conservative part

  • The gold-standard database (PostgreSQL — free, open, ~30 years mature, runs anywhere), with every structural change versioned and replayable like code.
  • Backups with a rehearsed restore. Ready-made backup/restore commands, point-in-time recovery guidance, and a disaster drill that was actually performed. An untested backup plan is a hope, not a plan.
  • Typo-proof data access. The database’s structure is wired into the code’s safety net — asking for a field that doesn’t exist is flagged as it’s typed, not discovered by a customer.
  • A professionally built example feature to copy — pagination that stays fast at page 400, all-or-nothing multi-step changes, instant-feeling updates — so the team’s first real feature inherits professional standards.

Feature value · Speed

Fast by default — and guarded against getting slow

  • Pages answer instantly. The newest rendering machinery is on by default: the static shell of every page is pre-built and served immediately, personalized parts stream in — plus an automatic code optimizer engineers used to hand-tune. Speed is the out-of-the-box posture, not a later project.
  • Interactions feel immediate. Screens update optimistically while the server confirms in the background, rolling back gracefully on failure.
  • Slowness can’t creep in silently. Hard size budgets on what browsers must download turn “the app got slower” into a specific failed check on the specific change that caused it.
  • Why speed is money: page speed measurably moves conversion, retention, and search ranking. Amazon-class teams treat 100ms as revenue.

Feature value · Reach

More markets, more users, less legal risk

  • International expansion becomes a translation purchase, not a re-engineering project. Multi-language support is built into the foundation (English + Spanish working today), with per-language web addresses that search engines index separately — retrofitting this later is one of software’s most notorious rebuilds.
  • Dates, numbers and currency format themselves per region — the small details that make software feel foreign when wrong.
  • Accessibility is machine-enforced, not aspirational. The industry-standard checker scans seven app surfaces on every change and fails the build on violations; components are built on a foundation engineered for keyboard and screen-reader correctness.
  • Why that’s defensibility, not polish: accessibility is now law across much of the market (ADA, European Accessibility Act), a hard requirement in government/enterprise purchasing — and lawsuits are routine.

Feature value · Look & feel

A product that looks professional — and can’t silently stop

  • One design system, enforced by construction. Approved colors, spacing, and components are the path of least resistance — no drift into seventeen shades of blue as the team grows.
  • The component catalog is owned, not rented. Every button and dialog’s source lives in the repository — customization is just editing; no upstream vendor can change or abandon the product’s look.
  • Dark mode done right — including the classic “flash of the wrong theme” bug, specifically engineered away. Table stakes for modern software; fiddly to retrofit.
  • Appearance is regression-tested. Every component is photographed in both themes and pixel-compared on every change — “we fixed the menu and broke the checkout button” gets caught by machines, plus a component showroom for designer review.

Feature value · Security

Hardened at the front door and at the loading dock

  • Front door: the full set of browser-enforced protections (content security policy and friends) against the standard web attack catalog, bot protection, attempt throttling, plus a standardized channel for security researchers to report responsibly.
  • Loading dock — the supply chain: modern apps are assembled from hundreds of open-source packages, and criminals hijack popular ones. Defenses here: a 7-day quarantine on all new package versions (enforced at two independent layers), an update robot whose every suggestion must pass the full test gauntlet, vulnerability scanning of code and shipping containers, and deep static analysis on every change.
  • Procurement-grade evidence: every build publishes a signed ingredients list (SBOM) and cryptographic proof of where it was built (SLSA provenance) — increasingly demanded by enterprise and government buyers, almost never found in starter kits.
  • Honest trade-offs: even the one place the gold-standard hardening conflicts with the speed machinery is documented, priced, and shipped as a verified opt-in recipe — not silently ignored.

Feature value · Quality

Quality is infrastructure here, not heroics

  • Every change runs the gauntlet before it can land: hundreds of unit tests, robot-driven real-browser journeys (sign-up through two-factor and passkey flows), database integration tests, accessibility scans, screenshot comparisons — and the deployable containers themselves are rebuilt, booted, health-checked, and security-scanned.
  • Quality only ratchets up. Test-coverage thresholds fail the build if protection drops; a dead-code detector blocks accumulation of abandoned code (it caught two real defects on day one).
  • Sustainable velocity is the payoff. Teams with this net ship on Friday afternoons; code review becomes judgment (“is this the right approach?”) instead of archaeology (“does this even work?”).
  • And it’s what makes AI participation safe — the same machines that catch careless humans catch careless AI, with zero sentiment. Verification, not trust.

Feature value · Freedom to run anywhere

No vendor can hold this product hostage

  • Ships in the universal container format (Docker) — runs identically on any cloud, any host, or your own hardware. Most kits in this ecosystem quietly assume one premium host; this one deliberately doesn’t.
  • The path is proven, not theoretical: a real copy was deployed live (Fly.io, managed database, healthy end-to-end) and the step-by-step runbook from that deployment ships in the docs — alternatives documented too.
  • Evaluation costs minutes: boots with two settings; all ~12 integrations are optional and switch off gracefully when unconfigured — a property enforced by the test suite, and the same design that makes every vendor swappable (each integration even ships a “how to remove this” checklist).
  • Misconfiguration fails loud and early — settings are validated at startup with clear errors, and secrets are structurally kept out of the code, with GitHub’s leak-blocking as a second net.
  • Even monitoring isn’t locked in: one optional setting exports performance traces in OpenTelemetry — the industry’s vendor-neutral monitoring language — to any compatible product or a collector you host yourself.

Feature value · Operations

Know about problems before customers do

  • Every crash reported with full context — exact line, user’s environment, steps leading in. Without this, you learn about bugs from the minority of users who complain; the rest just leave.
  • Uptime, logs, and a worker heartbeat — including the component whose silent death nobody notices until emails stop. Monitoring configuration is versioned as code, not hand-clicked and forgotten.
  • Product analytics with feature flags — see what’s used, roll features out gradually, kill a bad one instantly without redeploying.
  • Privacy-first by default: analytics is consent-gated (off until opt-in) and a user data export ships — GDPR alignment built in, and a deliberate choice of analytics vendor over Google Analytics on exactly these grounds.

Differentiator 3 · How it was built

This is what agentic development looks like when it’s done with discipline

2021 — Autocomplete
AI suggests the next few lines while a human writes. The human does all the thinking.
2023 — Chatbots
AI writes snippets on request, but can’t see the project, run anything, or check its work.
2025 — Agents
AI performs the work: reads the whole project, plans multi-step changes, writes the code, runs the builds and tests, observes real failures, and fixes them — under human direction. This project is the exhibit: built almost entirely this way, to the quality bar shown on every previous slide.

The human role shifts up, not out: setting direction, reviewing plans, approving trade-offs, owning judgment — an engineering manager of a tireless, extremely fast, occasionally overconfident staff engineer.

Differentiator 3 · Why it’s safe

The productivity isn’t free — it’s purchased with discipline

  • Plan → sign-off → build. The agent must present a plan and stop; a human approves before any code is written. Judgment stays human.
  • Verify by running. Every “done” is machine-checked evidence — full quality gate green plus the feature observed working live — never the AI’s say-so.
  • Context engineering. A standardized onboarding file plus 14 focused per-domain docs brief the agent precisely, per task. Because an agent remembers nothing between sessions, docs are load-bearing — updated with every change, audited for drift. Happy side effect: the best new-hire onboarding money can buy.
  • Guardrails, not trust. A committed permissions allowlist plus every quality machine from the previous slides, applied to AI output with zero sentiment.
  • The transferable claim: “agent-ready” is becoming a property of codebases, like “well-tested” — and this repository is ahead of that curve.

Honest boundaries

What it is not — the caveats that make the pitch credible

  • Not a finished product. It’s the foundation plus worked examples; the actual business features still get built on top.
  • Not a no-code tool. It makes engineers (human and AI) dramatically faster; it doesn’t replace them.
  • Opinionated, with receipts. One good tool per category, deeply integrated — and every pick argued in writing, every rejection recorded, every integration removable by checklist.
  • Known imperfections are public. A short backlog of deferred items exists, each with its reason — most gated on outside technology maturing.
  • The audits are self-run — rigorous and evidence-based, but not an external certification. The claims that matter most are the independently checkable ones: the dated verification log and the public code.
